# Setting up atlas on a new machine

atlas is local-first. The data store (`~/.atlas`) is **regenerable** — you never migrate it.
To stand atlas up on a new laptop you bring two things from git (this tool repo + your target
repos) and regenerate the rest by re-scanning. The map is a cheap-to-regenerate hint
(philosophy #5), so **re-scanning *is* the migration.**

## Prerequisites
- **Node ≥ 18** (developed on 22).
- A **C toolchain** for the tree-sitter native grammars (Swift/Kotlin/Go extractors compile via
  node-gyp on install):
  - macOS: `xcode-select --install`
  - Debian/Ubuntu: `sudo apt-get install build-essential python3`
- Optional: **Claude Code**, if you want the MCP/steering integration.

## 1. Get the tool repo
atlas lives in git — push it to a remote first if you haven't, then on the new machine:
```bash
git clone <your-atlas-remote> atlas && cd atlas
npm install        # .npmrc sets legacy-peer-deps; tree-sitter grammars build here
npm run build
npm test           # sanity: should be all green
```
Optional convenience: `npm link` (gives you a global `atlas`), or
`alias atlas='node /abs/path/to/atlas/bin/atlas.js'`.

## 2. Check out your target repos
Clone the code you analyze (e.g. ghost_daddy, the HMS repos) anywhere on disk. Their paths will
differ from the old machine — that's expected; you re-scan with the new paths.

## 3. Recreate the workspaces (rebuilds `~/.atlas`)
Re-scan each repo into its workspace. This rebuilds the manifests with the **new local paths**
and regenerates topology + the cross-repo map. Override the store root with `ATLAS_HOME` if you
don't want `~/.atlas`.
```bash
atlas scan /new/path/ghost_daddy --id ghost-daddy -w ghost
atlas scan /new/path/HMS/hms-admin    --id hms-admin    -w hms
atlas scan /new/path/HMS/hms-backend  --id hms-backend  -w hms
# … one scan per repo (role/language auto-detected)
atlas status        # confirm what's registered
```

## 4. Generate the agent docs
```bash
atlas agent -w ghost
atlas agent -w hms
```

## 5. Re-wire agents (machine-local — must be redone)
These point at absolute paths / live in per-machine config, so they don't travel:
- **MCP server** (user scope = available in every project):
  ```bash
  claude mcp add atlas -s user -- node /abs/path/to/atlas/bin/atlas.js mcp
  ```
- **Steering import** in each target repo's `CLAUDE.md`: the `@<path>` points at
  `~/.atlas/<ws>/atlas.steering.md`. If your username/home differs from the old machine, fix the
  path — `atlas agent -w <ws>` prints the exact line to use.
- **Auto-refresh + search-nudge hooks** (`.git/hooks`, `.claude/settings.json` are per-machine):
  ```bash
  atlas hook install      /new/path/repo -w <ws>
  atlas hook search-nudge /new/path/repo -w <ws>
  ```

## What you do NOT copy
- `~/.atlas/*` — regenerated by `scan` / `refresh` / `agent`.
- `dist/` — rebuilt by `npm run build`.
- `node_modules/` — reinstalled by `npm install`.

## Keeping machines in sync
Push the atlas tool repo and your target repos to git. On any machine: clone → `npm install` →
`npm run build` → re-scan. Done. (For a true one-shot, `atlas refresh -w <ws>` after the repos
are checked out re-scans every repo already in a workspace manifest — but on a fresh machine the
manifest doesn't exist yet, so the first time you scan each repo as in step 3.)
