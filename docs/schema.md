# Schema — the keystone data contract

> **Schema version: 0.** Treat this like a public API. Version it, changelog every change,
> never break it silently. Everything else in the codebase may churn; this contract is the
> stable seam between extractors and core. See [philosophy](philosophy.md) principles 8 & 9.

There are three documents in the system. Extractors produce (2). The core consumes (2) and
produces (3). The manifest (1) is hand-written or generated by `atlas scan`.

---

## 1. Manifest — defines scope

One per workspace. Lives in the central store at `~/.atlas/<workspace>/manifest.json`.

```json
{
  "schemaVersion": 0,
  "workspace": "client-acme",
  "type": "freelance",
  "repos": [
    {
      "id": "acme-web",
      "path": "/abs/path/to/acme-web",
      "role": "fe",
      "language": "typescript"
    }
  ]
}
```

- `type`: `freelance` | `company`
- `role`: `fe` | `be` | `lib` | `tool`
- `path` points anywhere on disk — repos are never copied or modified.

---

## 2. Extractor output — one per repo (the normalized form)

Every language extractor, regardless of language, emits exactly this shape. The core never
sees source code, only this.

```json
{
  "schemaVersion": 0,
  "repo": "acme-web",
  "generatedAt": "2026-06-04T00:00:00Z",
  "nodes": [
    {
      "id": "acme-web:src/orders/api.ts#createOrder",
      "kind": "function",
      "name": "createOrder",
      "file": "src/orders/api.ts",
      "line": 12
    }
  ],
  "edges": [
    {
      "from": "acme-web:src/orders/api.ts#createOrder",
      "to": "acme-web:src/lib/http.ts#post",
      "kind": "call",
      "line": 15
    }
  ],
  "endpoints": {
    "consumes": [
      { "method": "POST", "path": "/api/orders", "from": "acme-web:src/orders/api.ts#createOrder", "line": 16 }
    ],
    "exposes": [
      { "method": "GET", "path": "/api/orders/:id", "handler": "orders-svc:...#getOrder", "line": 40 }
    ]
  }
}
```

**Node `kind`:** `function` | `class` | `module` | `service` | `endpoint` | `external`
**Edge `kind`:** `call` | `import` | `inherit` | `expose` | `consume`
**Node `id` format:** `<repoId>:<relativeFile>#<symbol>` — globally unique across the workspace.
A whole-file (`module`) node has no symbol, so its id omits the suffix:
`<repoId>:<relativeFile>`. `import` edges connect these module nodes (file → file).

**Consume `path` may be symbolic.** When a client call's path is statically resolvable it's
a real route (`/api/orders/:id`); when it's indirected through a registry/helper the extractor
records the expression verbatim (e.g. `resolveSlug("auth","register")`). The core treats only
real routes as matchable — symbolic paths never link and stay `external` (principle #4/#5).
Whether a path is matchable is derived by the core, not stored, so the shape above is unchanged.

---

## 3. Merged map — produced by the core

The core merges all extractor outputs and resolves cross-repo links by **HTTP contract**
(`method` + normalized `path`, with param names stripped: `/api/orders/:id` → `/api/orders/{}`).

```json
{
  "schemaVersion": 0,
  "workspace": "client-acme",
  "generatedAt": "2026-06-04T00:00:00Z",
  "repos": ["acme-web", "orders-svc"],
  "crossRepoEdges": [
    {
      "from": "acme-web:src/orders/api.ts#createOrder",
      "to": "orders-svc:src/handlers/orders.ts#createOrderHandler",
      "kind": "http",
      "contract": "POST /api/orders"
    }
  ],
  "externalNodes": [
    {
      "id": "external:POST /api/payments",
      "reason": "consumed by acme-web but no repo in the manifest exposes it",
      "consumedBy": ["acme-web:src/billing/pay.ts#charge"]
    }
  ]
}
```

### The degradation rule (principle 4)
- `consumes` with a matching `exposes` in the manifest → a resolved `crossRepoEdge`.
- `consumes` with **no** match → an `external` node (NOT an error). If that repo is later
  added to the manifest, the next merge resolves it automatically.
- `exposes` with no consumer → recorded; may indicate an unused endpoint (do not flag as
  dead in early phases — that's a Phase 5 audit, and a hint, not a verdict).

---

## Evolving the schema

Bump `schemaVersion`, add a `CHANGELOG.md` entry, and prefer additive changes. If a breaking
change is unavoidable, write an ADR explaining why and how old data migrates.
